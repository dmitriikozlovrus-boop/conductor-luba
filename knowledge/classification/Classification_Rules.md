# Classification Rules

Version: 1.1

## Purpose

Данный документ определяет правила классификации входящей информации между сущностями системы.

Цель документа — обеспечить единообразную классификацию данных независимо от источника ввода, агента или модели.

При конфликте между несколькими сущностями необходимо руководствоваться данным документом.

---

# General Principles

## Principle 1. One Primary Entity

Каждая запись должна иметь одну основную сущность.

Дополнительные сущности могут быть связаны через отношения, но не должны подменять основную сущность.

## Principle 2. Classify by Primary Intent

Классификация производится по основному объекту управления.

Не по форме текста.
Не по используемым словам.
Не по наличию даты.
Не по наличию вопросительного знака.

А по тому, чем является запись по своей сути.

## Principle 3. Questions Create Study

Если основной целью является получение ответа на вопрос, создается Study.

## Principle 4. Actions Create Task

Если основной целью является выполнение действия, создается Task.

## Principle 5. Time Creates Event

Если основной целью является управление временем или резервирование временного слота, создается Event.

## Principle 6. Problems Are Not Solutions

Problem фиксирует нежелательное состояние.

Problem не содержит решение.
Problem не является задачей.

## Principle 7. Ideas Have No Commitment

Idea существует до момента принятия решения о реализации.

После принятия решения могут появляться Project, Task, Study или Improvement.

# Task vs Problem

Если основным объектом является действие → Task.

Если основным объектом является нежелательное состояние → Problem.

---

# Task vs Study

Если нужен ответ на вопрос → Study.

Если нужно выполнить действие → Task.

---

# Study vs Problem

Если основной объект — вопрос → Study.

Если основной объект — нежелательное состояние → Problem.

---

# Problem vs System Issue

Если проблема существует во внешнем мире пользователя, бизнеса, проекта или процесса → Problem.

Если проблема существует внутри системы, агента, базы данных, интеграции, памяти или инструкции → System Issue.

---

# Idea vs Study

Если существует вопрос для исследования → Study.

Если существует потенциально полезная мысль → Idea.

---

# Idea vs Task

Если необходимо выполнить действие → Task.

Если существует мысль без обязательства действовать → Idea.

---

# Idea vs Improvement

Если предложение относится к улучшению существующей системы → Improvement.

Если мысль не привязана к существующей системе → Idea.

---

# Improvement vs Task

Improvement описывает желаемое изменение системы.

Task описывает действие по реализации изменения.

---

# System Issue vs Improvement

System Issue описывает существующую проблему системы.

Improvement описывает предлагаемое улучшение системы.

---

# Event vs Task

Если основной объект — действие → Task.

Если основной объект — резервирование времени → Event.

Наличие даты или времени само по себе не делает запись Event.

---

# Event vs Communication

Event существует до или во время взаимодействия.

Communication существует после появления содержания взаимодействия.

---

# Communication vs Task

Communication фиксирует взаимодействие.

Task фиксирует действие.

---

# Project vs Task

Project объединяет несколько сущностей для достижения результата.

Task является отдельным действием.

---

# Project vs Stream

Stream отвечает на вопрос:

«За какую сферу жизни отвечает деятельность?»

Project отвечает на вопрос:

«Какой конкретный результат нужно получить?»

---

# Communication vs Contact

Contact — человек или организация.

Communication — факт взаимодействия с ними.

---

# Book vs Task

Книга является объектом библиотеки.

Задача является действием.

---

# Film vs Task

Фильм является объектом медиатеки.

Задача является действием.

---

# Goods vs Task

Товар является объектом учета.

Покупка товара является действием.

---

# Topic vs Project

Topic отвечает на вопрос:

«О чем идет речь?»

Project отвечает на вопрос:

«Какого результата необходимо достичь?»

---

# Entity Hierarchy

Stream
└── Project
    ├── Problem
    ├── Study
    ├── Task
    ├── Event
    ├── Communication
    ├── Idea
    ├── Improvement
    └── System Issue

Topic может быть связан с любой сущностью.

# Confidence Rules

Confidence ≥ 90%
→ создавать автоматически

Confidence 70–89%
→ создавать и помечать для проверки

Confidence < 70%
→ запрашивать уточнение

# Future Rule

При появлении нового спорного кейса:

1. Не изменять определения сущностей.
2. Добавлять новый пример в данный документ.
3. При накоплении кейсов создавать новый подраздел.
