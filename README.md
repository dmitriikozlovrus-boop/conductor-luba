# Conductor / Дирижер MVP

Минимальный сервис для Telegram-бота "Люба": принимает текст или голос, классифицирует поток на задачи и вопросы на изучение, записывает результат в Notion, а Todoist держит как следующий подключаемый слой.

## Что уже поддержано

- Telegram webhook: `POST /telegram/webhook`
- Проверка здоровья: `GET /healthz`
- Текстовые сообщения
- Голосовые/аудио сообщения через Telegram file API + OpenAI transcription
- AI-классификация на задачи и вопросы на изучение
- Уточнения в Telegram, если не хватает проекта, срока или уверенность ниже порога
- Создание задач в Notion `Tasks`
- Создание записей в Notion `Study / На изучение`
- Локальное хранение ожидающих уточнений в `data/pending.json`
- Полная двусторонняя синхронизация базы `TASKS` и Todoist

## Быстрый старт

1. Создай `.env` из примера:

```bash
cp .env.example .env
```

2. Заполни токены:

- `TELEGRAM_BOT_TOKEN`
- `OPENAI_API_KEY`
- `NOTION_TOKEN`

3. Запусти локально:

```bash
python3 -m conductor.app
```

4. Для локального теста без Telegram:

```bash
python3 -m conductor.cli "Завтра напомни написать Марко по алюминию. И изучить доступные логистические пути в Веракрус"
```

## Важная логика MVP

- Если срок не указан, Дирижер спрашивает срок.
- Если проект не найден или уверенность ниже `CONFIDENCE_THRESHOLD`, Дирижер спрашивает уточнение.
- Если в сообщении есть и задачи, и вопросы на изучение, создаются обе сущности.
- Исходный RAW отдельно не сохраняется.
- Todoist включается при наличии `TODOIST_API_TOKEN`; аварийная пауза управляется
  только переменной `TODOIST_SYNC_PAUSED`.

## TASKS ↔ Todoist

После включения сервис каждые `TODOIST_SYNC_INTERVAL_SECONDS` выполняет reconciliation:

- новая активная задача Notion создаётся в Todoist;
- новая активная задача Todoist создаётся в Notion;
- при первом запуске импортируется полная доступная история завершённых задач Todoist,
  затем каждая сверка загружает завершения за последние 7 дней;
- завершённые и отменённые задачи Notion без связи с Todoist не создаются в Todoist повторно;
- связанная активная задача, удалённая из Todoist, переводится в `Cancelled` в Notion и не воскресает;
- изменения названия, описания, срока, дедлайна, приоритета, проекта и Stream передаются в обе стороны;
- `Done` в Notion завершает задачу Todoist;
- `Cancelled` в Notion завершает задачу Todoist, сохраняя историю;
- завершение, восстановление и удаление из Todoist передаются в Notion через webhook;
- если изменилась только одна сторона, её версия переносится на вторую;
- если обе стороны изменились, побеждает более позднее изменение; при равном времени побеждает Todoist;
- после потери локальной истории первым безопасным снимком считается текущая версия Todoist;
- локальные отпечатки не позволяют изменениям циклически ходить между системами.

Обязательные поля базы Notion: `Task`, `Описание`, `Статус`, `Deadline`, `Срок выполнения`, `Strategic Impact`,
`Source`, `Todoist ID`, `Sync status`, `Sync error`.

Сопоставление полей:

| Notion `TASKS` | Todoist |
| --- | --- |
| `Task` | название задачи |
| `Описание` | описание задачи |
| `Статус: Done` | задача завершена |
| `Статус: Cancelled` | задача завершена с сохранением истории |
| `Срок выполнения` | due date |
| `Deadline` | deadline |
| `Strategic Impact` | priority |
| `Проект` | единственная управляемая метка задачи |
| `Todoist ID` | стабильная связь записей |
| `Sync status` | техническое состояние синхронизации |
| `Sync error` | последняя причина ошибки; очищается после успешной сверки |

После свежего деплоя или потери локальной истории синхронизации текущая
раскладка задач по разделам Todoist сначала импортируется в `TASKS.Stream`.
Это позволяет один раз разложить задачи в Todoist и не потерять разметку при
следующем обновлении сервиса.

Notion является источником каталога доступных проектов, а Todoist — основным интерфейсом назначения проекта задаче:

- каждая запись базы `PROJECTS` автоматически получает одноимённую метку в Todoist;
- метка проекта в Todoist записывается в `TASKS.Проект`;
- ручное изменение `TASKS.Проект` записывается обратно в метку Todoist;
- неизвестная метка Todoist не создаёт новый проект автоматически.

Лишние метки не удаляются из общего каталога Todoist автоматически, чтобы не уничтожать пользовательские
данные, но они не влияют на Notion и снимаются с синхронизируемых активных задач.

Все активные задачи Todoist хранятся во `Входящих` и распределяются по разделам из базы
`STREAMS`. Раздел Todoist и `TASKS.Stream` синхронизируются в обе стороны; если Stream пустой,
задача попадает в раздел `ПРОЧЕЕ`.

`Stream` задачи соответствует разделу внутри Todoist Inbox; задачи без Stream попадают в `ПРОЧЕЕ`.
Связь `Проект` соответствует единственной метке Todoist. Остальные поля без прямого аналога Todoist
(`Context`, `Энергия`, `Importance`, `Source`) остаются источником истины в Notion.

Если задаче в Todoist назначена метка проекта, а сама задача находится без раздела или в `ПРОЧЕЕ`,
она автоматически перемещается в раздел Stream, связанного с этим проектом в Notion. Ручное
размещение задачи в одном из Stream-разделов имеет приоритет и автоматически не перезаписывается.

Todoist считается основным интерфейсом задач. Ручная сортировка в Todoist записывается в Notion.
Ручное изменение в Notion также записывается в Todoist, если оно сделано позже.

Для запуска:

```bash
TODOIST_SYNC_PAUSED=false
TODOIST_API_TOKEN=...
TODOIST_WEBHOOK_SECRET=...
TASK_SYNC_SECRET=...
```

Аварийная пауза без удаления токенов:

```bash
TODOIST_SYNC_PAUSED=true
```

Ручной reconciliation:

```bash
curl -X POST \
  -H "X-Conductor-Sync-Secret: $TASK_SYNC_SECRET" \
  https://YOUR_DOMAIN/tasks/sync
```

Для мгновенного Todoist → Notion обновления зарегистрируй webhook приложения Todoist:

```text
https://YOUR_DOMAIN/todoist/webhook
```

В `TODOIST_WEBHOOK_SECRET` указывается Client Secret приложения Todoist. Без webhook периодический
reconciliation продолжает синхронизировать активные задачи и завершения. Удаления определяются
периодической сверкой после успешной загрузки истории завершённых задач. С webhook эти изменения
попадают в Notion почти сразу.

## Telegram webhook

Для публичного запуска нужен HTTPS URL. После деплоя выстави webhook:

```bash
curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook?url=https://YOUR_DOMAIN/telegram/webhook"
```

Или через скрипт:

```bash
TELEGRAM_BOT_TOKEN=... PUBLIC_BASE_URL=https://YOUR_DOMAIN sh deploy/set_webhook.sh
```

## Онлайн-запуск

Проект подготовлен для Render через `Dockerfile` и `render.yaml`.

Нужные переменные окружения на хостинге:

- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_WEBHOOK_SECRET` — любая длинная случайная строка, опционально, но желательно
- `OPENAI_API_KEY`
- `NOTION_TOKEN`
- `NOTION_TASKS_DATABASE_ID`
- `NOTION_STUDY_DATABASE_ID`
- `NOTION_PROJECTS_DATABASE_ID`
- `TODOIST_API_TOKEN` настроен
- `TODOIST_API_TOKEN`
- `TODOIST_WEBHOOK_SECRET`
- `TASK_SYNC_SECRET`

После деплоя надо вызвать Telegram `setWebhook` на публичный URL сервиса.
